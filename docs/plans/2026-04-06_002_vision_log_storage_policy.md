# Plan

## Goal
비전 기반 외부 측정계의 로그 저장 경로, 파일명 규칙, run 단위 관리 방식, 메인 로그와의 후처리 병합 기준을 정의한다.
현재 프로젝트의 메인 CSV 계약은 유지하고, 비전 로그는 별도 저장 계층으로 분리한다.

## Files
- `docs/plans/2026-04-06_002_vision_log_storage_policy.md`
- `docs/vision_tracking.md`
- `docs/daily_notes/2026-04-06.md`

## Changes
- 비전 로그의 저장 위치를 프로젝트 구조 안에서 명시한다.
- 초기 단계 기준으로 비전 로그는 `raw` 중심으로 저장하는 정책을 정의한다.
- run 단위 식별자(`run_id`)와 파일명 규칙의 최소 기준을 정리한다.
- 비전 로그 CSV의 최소 필드와 선택 필드를 정리한다.
- 메인 로그와 비전 로그를 직접 혼합 저장하지 않고, 후처리 병합 대상으로 유지한다는 원칙을 명시한다.
- 병합 기준을 timestamp alignment 기반으로 정의하고, 어떤 로그가 기준 시간축이 되는지 정리한다.
- 실험 재현성을 위해 실험별 로그 묶음 구조를 문서화한다.

## Impact
- 메인 CSV 고정 컬럼 계약은 유지된다.
- 비전 로그는 외부 측정계 전용 데이터로 분리 관리된다.
- 실험 데이터셋 정리 방식이 명확해져 학습 및 검증 재현성이 높아진다.
- 향후 비전 기반 구현 코드가 따라야 할 저장 규칙의 기준이 생긴다.

## Risk
- `raw` 중심 정책만 먼저 정하면 가공 데이터 구조는 후속 결정이 필요하다.
- run_id 규칙이 불명확하면 메인 로그와 비전 로그의 대응 관계가 깨질 수 있다.
- 원본 영상과 CSV를 함께 저장할 경우 용량 관리 문제가 생길 수 있다.
- 메인 로그와 비전 로그의 시간축 기준이 다르면 후처리 병합 오류가 발생할 수 있다.

## Validation
- 저장 경로 정책이 현재 리포 구조(`data/`, `experiments/`)와 자연스럽게 맞는지 확인한다.
- 메인 CSV 계약을 바꾸지 않는다는 점이 유지되는지 확인한다.
- run 단위로 로그와 메타데이터를 재현 가능하게 추적할 수 있는지 확인한다.
- 비전 로그 최소 필드, 파일명 규칙, 병합 기준이 문서에 빠짐없이 포함되는지 확인한다.
